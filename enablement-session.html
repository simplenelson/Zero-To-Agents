Module 1Why Process Matters

Vibe Coding — What AI Owns vs. What You Own

✅ AI Handles This (the 80%)
  • →Code scaffolding from existing patterns
  • →Following project conventions (imports, @tool decorators, class structure)
  • →Generating boilerplate test files
  • →Writing commit messages and PR descriptions
  • →Explaining unfamiliar code
  • →Suggesting fixes for failing tests
⚠️ You Must Own This (the 20%)
  • !Business requirements — what the feature should do
  • !Reviewing generated code for correctness
  • !Security rules — secrets, auth, data access
  • !Environment configuration (sandbox vs. prod)
  • !Final approval before code merges
  • !Deciding when not to accept a Copilot suggestion

💡 The mental model: Copilot is a very fast junior developer who knows the codebase well but doesn't know your business. You're the senior engineer reviewing the pull request.

This is the most important mindset shift. Vibe coding works great when you stay in the driver's seat. Copilot will write correct-looking code that is wrong for your business. It doesn't know what AccountPulse should say, or what a JOLT email is, or that your rep IDs are in a specific format. YOU know that. The copilot-instructions.md file is how we encode that business knowledge into the AI's context.

Why Git Exists
  • →Snapshot history — every save point is recoverable

  • →Safe experimentation — try something on a branch; delete it if it breaks

  • ✓feature/fix-next-move-ranking
  • ✓feature/improve-email-draft-prompt

Lowercase · hyphens · under 50 chars

Think of branches like parallel universes. You do your work in your own universe (feature branch), and only merge it into the shared universe (dev) when it's ready. The confidence level increases as code moves right. You'd never push directly to main — that's guarded by approvals and automated tests.
Module 2Git Strategy

DAP Branch Strategy

Branch Purpose Who Touches It Protection Rules
feature/* Individual feature development Any team member None — experiment freely
dev Shared engineering integration — all features land here first Dev team lead PR + CI must pass · at least 1 reviewer
integration Non-prod sandbox deployment — auto-deploys to Azure Dev/infra coordination PR + CI must pass · optional infra reviewer gate
main Production — only approved, tested releases Infra team approves PR from integration only · required infra reviewers · no self-review
Separate Tracks (don't touch routine delivery)
bootstrap-foundation new Azure resources, VNet, one-time infra
teams-catalog-admin M365 app package publish (admin approval required)
The key mental model: feature/* is your playground, dev is the team's integration point, integration is the sandbox mirror of production, and main IS production. Never skip levels. If your feature branch merges directly to main, nobody had a chance to catch mistakes and the pipeline didn't validate it in sandbox first.
Module 2Git Strategy

Pull Requests: The Review Gate

PR Lifecycle
1
Create PR — push your branch, open PR to dev. Copilot auto-generates the description.
2
CI runs automatically — tests, Docker build, shell validation. No human needed for this step.
3
Copilot reviews — inline comments on patterns, potential bugs, convention violations.
4
Human reviewer approves — checks business logic, approves the merge.
5
Merge — code lands in dev. Feature branch can be deleted.
PRs are conversations, not just approvals

A PR is where the team discusses the change before it affects anyone. Comments are questions, suggestions, or concerns — not personal criticism.

What Copilot writes for you
  • →PR title and description from the diff
  • →Summary of files changed and why
  • →Inline suggestions on code quality
  • →Commit messages for each save point
PRs are how code gets reviewed before it's shared. You can't break anything by opening a PR — it's just a proposal. The CI checks run automatically, so you get fast feedback. Copilot's review is especially useful for catching things like "you forgot to register this agent in HandoffBuilder" or "this env var read belongs in config.py".
Module 3Automated Pipeline

GitHub Actions: Your Robot QA Team

What CI checks on every PR
  • ✓Python unit tests (pytest mvp/agents/tests/)
  • ✓Wrapper tests (pytest mvp/m365_wrapper/tests/)
  • ✓Docker build smoke — both containers build successfully
  • ✓Shell script validation — infra scripts have no bash errors
  • ✓M365 app package build

🤖 All of this runs automatically — no human presses any button. Red = something is broken. Green = safe to review and merge.

The 4 Workflows
WorkflowWhenWhat it does
ci.yml Every PR / push Tests, builds, validates
deploy-integration.yml Push to integration Deploy to sandbox, run smoke tests
deploy-production.yml Push to main Promote tested image to prod
publish-teams-catalog.yml Manual (M365 admin) Update Teams app package

🔐 Authentication uses Azure OIDC — no stored passwords or API keys in GitHub. Separate identities for integration vs. production.

GitHub Actions is the automation layer. Every time code is pushed, these workflows fire. The CI workflow is the gatekeeper — if tests fail, the PR is blocked. Nobody has to remember to run tests manually. The OIDC auth model is worth highlighting: no secrets stored in GitHub for Azure access. Instead, GitHub's identity is federated with Azure Entra so it can authenticate dynamically.
Module 3Automated Pipeline

What Runs & When

On every PR
  • ✓ Unit tests (agents · wrapper · infra)
  • ✓ Docker build smoke (local, no push)
  • ✓ Shell script syntax check
On merge → integration
  • ✓ Build & push both images to ACR
  • ✓ Deploy to Container Apps (sandbox)
  • ✓ Smoke tests (health · auth · chat)
On merge → main
  • ✓ Human approval required
  • ✓ Same ACR image promoted to prod (no rebuild)
Three columns, three moments in the lifecycle. PRs get fast feedback — tests and a local Docker build. Nothing touches Azure until the integration branch merge, where images are built in ACR, deployed to sandbox, and smoke-tested. Production is just a promotion of what already passed — no new build, no surprises.
Module 4GitHub Copilot

Building a New Agent with Copilot

💬 Ask

Start by asking Copilot to understand the codebase before writing anything.

  • "How does account_pulse.py register its tools?"
  • "What pattern does the planner use to call specialist agents?"
  • "Which files would I touch to add a new agent?"

Use @workspace to search the whole repo or #file to pin a specific file.

📋 Plan

Ask Copilot to produce a plan before generating any code.

  • "Give me a step-by-step plan to add a Renewal Risk agent following the same structure as AccountPulse."
  • "What Databricks queries will I need and where should they live?"

Review the plan, correct it, then proceed — cheaper to fix intent than code.

🤖 Agent Mode

Hand the approved plan to Agent Mode and let it execute across the repo.

  • Creates the new agent file with correct @tool decorators
  • Registers it in planner.py and __init__.py
  • Adds unit tests following the existing test patterns

Agent Mode reads context, edits multiple files, and checks its own work.

This is the practical workflow the team should use. Don't jump straight to Agent Mode — start by asking questions so Copilot understands the conventions. Then get a written plan you can review. Only then let Agent Mode loose across multiple files. The Ask → Plan → Agent flow means Copilot follows your existing patterns rather than inventing its own. The codebase is the spec.

Agent Mode: Multi-File Changes

Chat vs Agent Mode
💬
Chat — Question & Answer. Explains, suggests, shows code snippets. You copy and paste.
🤖
Agent Mode — Plans and executes. Reads multiple files, creates new files, edits existing ones. You review and accept/reject.
When to use Agent Mode for DAP
  • →Creating a new specialist agent end-to-end
  • →Adding a new Databricks-backed tool + tests
  • →Refactoring a module to follow updated conventions
  • →Generating a test file for existing code
Example Agent Mode Prompt
Using the existing patterns in account_pulse.py and next_move.py, create a new specialist agent called CompetitorIntel that returns a list of competitor mentions for the seller's top accounts. Follow ALL conventions in .github/copilot-instructions.md: - try/except import pattern - @tool pattern for data access - no SQL in the agent - register in planner.py HandoffBuilder - add unit tests in agents/tests/
Agent will: read 5+ files → create competitor_intel.py → update planner.py → create test file
DEMO MOMENT: Switch to Agent Mode. Paste the example prompt. Let it run. Walk through what it reads and creates. Highlight: (1) it follows the try/except import pattern automatically, (2) it registers in HandoffBuilder without being told explicitly because copilot-instructions.md says to, (3) it creates a test file. Then point at one thing to review — maybe the agent instructions string needs real business content that only you know.
Module 4GitHub Copilot

Prompting Well — The 3-Part Formula

1. Context

Tell Copilot WHAT it's working with. Reference specific files, the instructions doc, the existing pattern to follow.

"Given account_pulse.py and copilot-instructions.md..."
2. Action

Tell Copilot WHAT to create or change. Be specific — vague requests get vague outputs.

"...create a CompetitorIntel agent that returns competitor mentions for accounts..."
3. Constraints

Tell Copilot what NOT to do or what rules to follow. This is where DAP-specific guardrails go.

"...no SQL in the agent, register in HandoffBuilder, follow all conventions in copilot-instructions.md"
📄

.github/copilot-instructions.md is your team's AI contract. It encodes every architectural rule, naming convention, and security guardrail into the context Copilot reads for every request in this repo. The more detailed and accurate it is, the better Copilot follows your patterns — automatically, without you repeating the rules every time.

The 3-part structure applies whether you're using Chat or Agent Mode. Bad prompt: "add a new agent". Good prompt: Context (which files to use as pattern) + Action (what the agent should do specifically) + Constraints (which rules to follow — and always reference copilot-instructions.md explicitly). The instructions file is the single most powerful lever the team has. If a Copilot output is wrong, the fix is usually to improve the instructions file, not to retry the prompt.
Module 4GitHub Copilot

Copilot for PRs + DAP Guardrails

PR Assistance Features
Auto-Generate PR Description

Click the Copilot sparkle on the PR description field → Copilot reads the diff and writes: what changed, which files were added/modified, why (inferred from code).

Inline Code Review

Copilot reviews your PR and leaves line-specific comments. Catches: missed function registration, missing tests, pattern deviations, potential bugs.

Top 5 DAP Guardrails for Copilot
  • ✗ No SQL in agents — if an agent needs data, it calls a @tool function. SQL never appears in prompts or agent instructions.
  • ✗ No Databricks from the M365 Wrapper — the wrapper is stateless and forwards only. All Databricks access lives in mvp/agents/.
  • ✗ No raw os.environ.get() in business logic — all env var reads go through config.py with @lru_cache.
  • ✗ New agents must register in HandoffBuilder — creating my_agent.py alone is not enough. Register in planner.py or it's never called.
  • ✗ Never scale planner past 1 replica — the in-memory session store is not distributed. ACA_MAX_REPLICAS must stay at 1.
DEMO MOMENT: After pushing the feature branch from the Agent Mode demo, go to GitHub, create the PR. Show the Copilot sparkle button on the description field. Click it and watch the auto-generated description appear. If time permits, show a Copilot code review comment. The guardrails slide is the most important safety content — these are the rules that Copilot might accidentally break if the prompt isn't explicit enough. They're all in copilot-instructions.md for this reason.
Module 5The Full Feature Loop

End-to-End: From Idea to Production

1
GitHub Issue — Write what you want to build. Be specific about business behavior, not implementation. Copilot can help draft the issue.
2
Create feature branch — git checkout -b feature/my-feature from dev. Never from main.
3
Agent Mode implementation — Use the 3-part prompt. Reference existing patterns and copilot-instructions.md. Review every file Copilot creates.
4
Run tests locally — uv run pytest mvp/agents/tests/. Fix failures before pushing. Copilot helps fix failing tests.
5
Commit with Copilot message — Click the sparkle in VS Code Source Control. Review and accept the generated commit message.
6
Push + open PR — git push origin feature/my-feature. Open PR to dev. Use Copilot to generate description.
7
CI runs automatically — Watch the checks tab. Red = fix before merge. Green = ready for review.
8
Copilot + human review — Copilot leaves inline comments. Reviewer approves. Address any feedback on the same branch.
9
Merge → dev → integration — Sandbox auto-deploys. Smoke tests validate the full runtime.
10
Infra approves → main → production — Same tested image promotes. Feature is live for sellers.
